Wendogo.

A multiplayer hidden-role card game. Players are survivors — and one of them is secretly the Wendogo. By day, cooperate and gather resources. By night, actions are hidden, and consequences only revealed at dawn. With real-time voice chat.

/ 01 — OVERVIEW

Half co-op. Half paranoia.

A modern take on hidden-role. The group needs each other to survive, but one of the group isn't really part of it. The clever bit isn't the accusation — it's the night, where actions are hidden and morning shows you only the consequences. Who did what is the puzzle.

Built over 3-4 months by a team of 10 (3 programmers), in Unity 6 for Android. I worked on the multiplayer networking, the voice chat integration, the day/night game loop, and the turn-by-turn logic for each player.

/ 05 — ARCHITECTURE

How it's built.

Multiplayer — Unity Netcode, server-authoritative

The networking stack is built on Unity Netcode for GameObjects. The architecture is server-authoritative: when a player plays a card, the action is sent to the server as a command. The server queries the state machine to check whether the transition is valid, applies the card's effect if so, updates the global state, and then broadcasts the result to all clients. Invalid actions are rejected and an error is returned to the sender only.

This keeps the game state consistent across all clients — each player always has a local state synchronized with the server's state machine, regardless of network conditions.

// Sequential diagram — playing a card: client → server → state machine → all clients

Architecture — PlayerController at the center

The PlayerController is the central class: it manages the player's state via a PlayerControllerSM, handles card confirmation, and connects to the ServerManager for session management. It also interacts with CardEffect (abstract card logic), CanvaTarget (UI highlighting), FXEventManager (visual feedback), and DeckConfiguration backed by a CardDatabaseSO ScriptableObject.

Session lifecycle is handled by AutoSessionBootstrapper and SessionManager, with SessionEventDispatcher notifying the rest of the system when players join, leave, or the session ends. Both GameStateMachine and PlayerControllerSM inherit from a shared StateMachine base class.

// Class diagram — PlayerController and its dependencies

Day / night game loop & turn order

Each round is split into two phases. During the day, players take turns sequentially (P1 → P2 → P3 → P4). Played cards are visible to everyone — this is the social phase, where discussion and bluffing happen. Once all players have played, card effects are resolved and applied collectively.

At night, the same turn order applies, but actions are hidden. Card effects are applied immediately as each player acts, but no one else sees what happened. Only at the next dawn are all night actions revealed — their consequences surfacing at once, fueling doubt and accusation for the next day.

Voice chat — Vivox

Real-time voice communication is handled via Vivox, integrated through Unity's built-in tooling. Voice chat is open during the day phase, where players can discuss, negotiate, and lie — and is a core part of the social deduction experience.
